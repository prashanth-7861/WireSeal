# wg-automate

WireGuard server automation CLI with zero plaintext secrets on disk. The tool manages key
generation, client lifecycle, firewall rules, and optional dynamic DNS -- all behind an
encrypted vault that never writes raw key material to disk. If the vault file is stolen
without the passphrase, no cryptographic material is exposed.

## Security Model

All WireGuard private keys and pre-shared keys are stored as `SecretBytes` objects in memory
and serialized only inside the AES-256-GCM vault. The vault key is derived from the user
passphrase using Argon2id (time_cost=3, memory_cost=262144, parallelism=4). Argon2 parameters
are stored in the vault header; the header is bound to the ciphertext as AES-GCM AAD so header
tampering invalidates the authentication tag. Audit log entries are written with no key material
(AUDIT-01). Passphrase input uses `click.prompt(hide_input=True)` exclusively -- never
environment variables or command-line flags (CLI-02).

## Threat Model

**Protected:** Vault file at rest (full disk encryption not assumed), WireGuard private keys,
pre-shared keys, dynamic DNS tokens stored in vault.

**Out of scope:** Memory forensics after wg-automate exits, kernel keyring attacks, a root
attacker who can ptrace the running process.

**Trust boundaries:** The user passphrase is the only secret that never touches disk. The vault
file is considered untrusted storage. Config files on disk (`/etc/wireguard/*.conf`) are
readable by root; protecting them from root is out of scope by design.

## Installation

**Option A: pip with hash verification (recommended for development)**

```bash
pip install --require-hashes -r requirements.txt
pip install -e .
```

All package hashes in `requirements.txt` are SHA-256 and generated by pip-compile. The
`--require-hashes` flag causes pip to reject any package whose hash does not match.

**Option B: standalone binary**

Download the binary for your platform from the GitHub Releases page. Verify integrity before
running (see Verifying a Release below).

## Verifying a Release

```bash
# 1. Download the binary and checksum file
curl -LO https://github.com/YOUR_ORG/wg-automate/releases/latest/download/wg-automate-linux-x86_64
curl -LO https://github.com/YOUR_ORG/wg-automate/releases/latest/download/sha256sums.txt

# 2. Verify the SHA-256 checksum
sha256sum -c sha256sums.txt --ignore-missing

# 3. Verify the Sigstore signature (keyless -- no stored private key)
# Install cosign: https://docs.sigstore.dev/cosign/installation/
cosign verify-blob \
  --certificate wg-automate-linux-x86_64.sigstore.json \
  --certificate-identity-regexp "https://github.com/YOUR_ORG/wg-automate" \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  wg-automate-linux-x86_64

# 4. Make executable and move to PATH
chmod +x wg-automate-linux-x86_64
sudo mv wg-automate-linux-x86_64 /usr/local/bin/wg-automate
```

## Quick Start

```bash
# Initialize the vault (creates ~/.wg-automate/vault.enc)
sudo wg-automate init --subnet 10.0.0.1/24 --port 51820 --interface wg0

# Add a client (generates keys, issues IP, writes peer config)
sudo wg-automate add-client alice

# Display the client config as a QR code for mobile import
sudo wg-automate show-qr alice

# Remove a client (revokes keys, removes peer, reloads WireGuard)
sudo wg-automate remove-client alice
```

## Commands Reference

| Command | Description |
|---|---|
| `init` | Initialize vault, generate server keypair, write server config |
| `status` | Show server public key, peer count, interface state |
| `verify` | Verify vault integrity (HMAC check without full unlock) |
| `lock` | Wipe in-memory vault state (noop if already locked) |
| `change-passphrase` | Re-encrypt vault under a new passphrase |
| `add-client` | Generate client keypair + PSK, assign IP, write peer config |
| `remove-client` | Revoke client keys, remove peer from server config |
| `list-clients` | Print all client names and assigned IPs |
| `show-qr` | Render client config as terminal QR code (times out in 60 s) |
| `export` | Export client config to stdout or file |
| `update-dns` | Push current public IP to DuckDNS dynamic DNS |
| `rotate-keys` | Generate new client keypair + PSK, reload WireGuard |
| `rotate-server-keys` | Generate new server keypair, update all peer configs |
| `audit-log` | Display audit log (no passphrase required) |

## Security Limitations

**Windows AV false positive:** The PyInstaller bootloader embedded in the standalone binary
may trigger Windows Defender heuristics. This is a known false positive from PyInstaller's
self-extracting executable format, not malicious code. Verify the Sigstore signature and
SHA-256 checksum (see Verifying a Release). If AV scanning is a hard requirement, build from
source or use pip install.

**mlock best-effort:** `SecretBytes` calls `mlock` to prevent swapping key material to disk.
`mlock` is best-effort and may fail silently if the process lacks `CAP_IPC_LOCK` (Linux) or
is running without elevated privileges. On systems where mlock fails, key material may be
written to swap.

**Python memory management:** Python's garbage collector does not guarantee immediate memory
reclamation. `SecretBytes.wipe()` zeroes the underlying bytearray before releasing the
reference, but Python may retain object headers or cached representations in memory briefly
after wipe. This is inherent to CPython and not specific to wg-automate.

## Contributing

Submit pull requests against the `main` branch. All commits must pass `pytest -m "not integration"`
before review. Integration tests (`pytest -m integration`) require a Docker daemon and run
automatically on merge to `main` in CI.

**Security issues:** Do not file public GitHub issues for security vulnerabilities. Send a
private disclosure to the maintainers. Include a minimal reproduction and the affected version.
Public disclosure is coordinated after a fix is available.
