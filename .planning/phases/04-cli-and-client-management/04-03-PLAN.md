--- rotate-keys (CLIENT-05) ---
@cli.command("rotate-keys")
@click.argument("name")
def rotate_keys(name):

The rotation must be atomic: new keys are generated and validated BEFORE old
keys are removed. The vault context manager is the commit boundary.

  1. Collect passphrase via click.prompt("Vault passphrase", hide_input=True).
  2. Open vault via Vault context manager.
  3. Verify name exists in vault state clients. Abort if not found.
  4. --- GENERATE NEW MATERIAL (do not touch old yet) ---
     Import generate_keypair from core.keygen; generate new_keypair.
     Import generate_psk from core.psk; generate new_psk.
  5. --- BUILD AND VALIDATE NEW CONFIGS ---
     Reconstruct new client config string using new keypair, new PSK,
     existing server public key, existing IP, server endpoint.
     Import ConfigBuilder; build updated server config replacing the old
     [Peer] block for name with a new [Peer] block using new client public key
     and new PSK.
     Import validator from security.validator; validate both configs.
     If validation fails: abort, wipe new_keypair and new_psk from memory,
     raise click.ClickException.
  6. --- WRITE NEW CONFIGS ATOMICALLY ---
     Write new client config to ~/.wg-automate/clients/{name}.conf via
     atomic_write (mode=0o600). Compute SHA-256; store new hash in vault state.
     Write updated server config atomically. Compute and store new server
     config hash.
  7. --- RELOAD WIREGUARD ---
     subprocess.run(wg syncconf command). If reload fails: the files on disk
     are already updated but WireGuard is using stale config. Print a warning:
     "WireGuard reload failed. Config on disk is updated but service may be
     stale. Run: wg syncconf wg0 <(wg-quick strip /path/to/wg0.conf)"
     Do NOT abort the rotation — the files are already correct.
  8. --- WIPE OLD KEYS ---
     Import wipe_bytes from security.secrets_wipe.
     Retrieve old private key SecretBytes from vault state and wipe it.
     Retrieve old PSK SecretBytes and wipe it.
     Update vault state clients[name] with new keypair and new PSK.
     (The vault context manager will atomically write this state on __exit__.)
  9. Log action="rotate-keys", name=name (no key material).
  10. Reconstruct new client config string (again from new vault state).
      Import generate_qr_terminal from core.qr_generator.
      Print new QR. Sleep 60 seconds. click.clear(). Wipe config string.
  11. Print "Keys rotated for client '{name}'. New QR displayed above."

ORDER MATTERS: Generate new -> Validate -> Write files -> Reload WireGuard ->
Wipe old -> Commit vault. This ensures old keys exist as fallback until the
new config is confirmed written to disk.

--- rotate-server-keys (CLIENT-06) ---
@cli.command("rotate-server-keys")
def rotate_server_keys():

Server key rotation is more complex: every client config must be updated with
the new server public key. All updates happen inside one vault context.

  1. Collect passphrase via click.prompt("Vault passphrase", hide_input=True).
     Print a confirmation warning: "This will regenerate the server keypair
     and update ALL client configs. All clients must reconnect. Proceed? [y/N]"
     Read response via click.confirm. Abort if not confirmed.
  2. Open vault via Vault context manager.
  3. Read client list from vault state (all client names).
  4. --- GENERATE NEW SERVER KEYPAIR ---
     Import generate_keypair from core.keygen; generate new_server_keypair.
  5. --- UPDATE ALL CLIENT CONFIGS ---
     For each client in vault state:
       Reconstruct client config with new server public key (same client keys,
       same IP, same PSK — only server public key changes).
       Validate client config. If any validation fails: wipe new_server_keypair,
       abort with error listing which client failed.
       Write updated client config atomically. Store new hash.
  6. --- UPDATE SERVER CONFIG ---
     Rebuild server config with new server private key. Keep all existing
     [Peer] blocks unchanged (client keys/PSKs are unchanged).
     Validate server config. Write atomically. Store new hash.
  7. --- RELOAD WIREGUARD ---
     wg syncconf. Same failure handling as rotate-keys step 7.
  8. --- WIPE OLD SERVER KEYS ---
     Wipe old server private key SecretBytes via wipe_bytes.
     Update vault state with new server keypair.
  9. Log action="rotate-server-keys", client_count=len(clients).
  10. Print "Server keypair rotated. {N} client config(s) updated.
      All clients must reconnect."

ORDER for server rotation: Generate new server keypair -> Update + validate
all client configs -> Update + validate server config -> Write all atomically ->
Reload WireGuard -> Wipe old server keys -> Commit vault.

--- audit-log (AUDIT-03) ---
@cli.command("audit-log")
@click.option("--lines", default=50, type=int, show_default=True,
              help="Number of recent entries to display")
def audit_log(lines):
  1. This command does NOT require vault unlock. The audit log is readable
     without the vault passphrase (it contains no secrets — AUDIT-01 ensures
     no key material is ever logged). This design means an admin can check
     the audit log without knowing the vault passphrase.
  2. Import AuditLog from security.audit.
     Instantiate AuditLog with the default log path.
     Call audit_log_instance.get_recent_entries(lines) to retrieve entries.
  3. If no entries: print "Audit log is empty."
  4. For each entry (NDJSON line parsed as dict):
     Format and print: "{timestamp}  [{action}]  {metadata}"
     Timestamp: ISO 8601 UTC, already in the entry.
     Action: e.g., "init", "add-client", "remove-client", "rotate-keys".
     Metadata: remaining fields except timestamp and action, formatted as
     key=value pairs. Skip any field named "error" if its value is None.
  5. Print divider line and total count: "-- {N} entries shown --"
  6. SECURITY INVARIANT: Verify no entry contains key material by asserting
     none of "PrivateKey", "psk", "passphrase", "token" appear as field names
     in any returned entry. If found: print a CRITICAL warning but do not
     crash (the invariant should have been enforced at write time by AUDIT-01).

The --lines option defaults to 50 (AUDIT-03) but allows larger values for
forensic review. No upper limit is enforced — the file is append-only and
bounded by disk space.
